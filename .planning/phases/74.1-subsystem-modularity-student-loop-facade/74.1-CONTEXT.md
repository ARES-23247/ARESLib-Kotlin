# Phase 74.1 Context: Subsystem Modularity & Student Loop Facade

## 1. The Core Problem

Novice students find traditional Redux patterns intimidating. To create or modify a subsystem (like a shooter or intake), they currently have to write or edit code in multiple disconnected layers:
1. **State**: Add state fields in `SuperstructureState.kt`.
2. **Action**: Declare an action case in `RobotAction.kt`.
3. **Reducer**: Handle the action and define state transition logic in `SuperstructureReducer.kt`.
4. **Hardware IO**: Map physical inputs and write outputs to motor controllers in `FtcRevHubIO.kt`.
5. **OpMode**: Manually instantiate the store, read from it, and call `store.dispatch(action)`.

This fragmentation makes onboarding difficult and increases cognitive load.

---

## 2. The Architectural Solution: Subsystem Facades + Robot Container

To provide a superior student experience while keeping our **100% pure, testable, and log-replayable Redux core**, we will implement a **Subsystem Facade Pattern** centered around a unified `AresRobot` container class.

```mermaid
graph TD
    subgraph Student Code (Linear OpMode)
        Op[AresHardwareTestOpMode] -->|robot.update| Robot[AresRobot]
        Op -->|robot.shooter.shoot| Robot
    end

    subgraph Facade Layer (Subsystem Views)
        Robot -->|delegates to| Shooter[ShooterSubsystem]
        Robot -->|delegates to| Intake[IntakeSubsystem]
    end

    subgraph Redux Core (Immutable & Testable)
        Shooter -->|dispatches SetFlywheelActive| Store[(Redux Store)]
        Intake -->|dispatches SetIntakePivot| Store
        Store -->|reads active state| Shooter
    end

    subgraph Hardware IO (HAL)
        Robot -->|automatically updates| IO[HardwareIO Layer]
    end
```

### Key Components

1. **`AresRobot` Container**:
   * Centrally owns the Redux `Store` and the hardware `IO` configurations.
   * Exposes high-level subsystem facades (e.g. `robot.shooter`, `robot.intake`, `robot.drive`).
   * Provides a single, clean `.update()` method that reads hardware inputs, runs the reducer loop, commands actuators, and streams telemetry automatically in the background.

2. **`Subsystem` Facade Classes**:
   * Each physical subsystem will be defined in a single file representing a vertical slice.
   * **Write Actions**: Exposes intuitive command methods (e.g. `cowl.setAngle(45.0)`, `shooter.shoot()`, `intake.deploy()`) which dispatch the appropriate Redux action under the hood.
   * **Read Properties**: Exposes clean read-only getters (e.g. `val flywheelRPM: Double get() = store.state.superstructure.flywheelRPM`) to fetch fields from the central state.

3. **Background Telemetry & AdvantageScope Logging**:
   * AdvantageScope logging and NT4 state telemetry are completely abstracted inside `AresRobot.update()`.
   * Students don't write any logging code; all fields, FSM states, and motor currents are automatically logged and published every frame.
